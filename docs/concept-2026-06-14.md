# persona-wire concept (P0 design doc)

Status: P0 land (initial concept consolidation)
Date: 2026-06-14
Authors: persona-wire contributors

## 1. Origin

Three concept thread converged:

| # | Layer contribution |
|---|---|
| 1 | **Concept base** — メタ Context Routing + Edge/Node Graph + Lifecycle + Knowledge SoT navigator |
| 2 | **Application surface** — 7 lifecycle/admin primitive (init / update / close / doctor / query / net / spawn / retire) |
| 3 | **Storage / engine** — 5 core primitive (Graph / CRUD / Compute / Prompt / AutoVersion) |

Naming note: "persona-network" was rejected because the "network" label leaks into UI/marketing scope confusion. "persona-wire" picked as working title to keep the concept neutral (synapse / graph / wire 等の仮称、 後で確定可)。

## 2. Core value proposition

**ProjectionAsPrompt** — arbitrary graph query → template binding → LLM-consumable Prompt (or Markdown / JSON / ASCII).

The graph is open-vocabulary (Node type / Edge type are registered, not hardcoded), so persona-wire serves as a **workflow engine / graph engine** that can host:
- Persona relation map (`routes_to` friend graph) seed
- Workflow definitions (step nodes + `transitions_to` edges + Constraint edges as guards)
- SoT cross-reference (`cites` / `derives_from`) across outline / mini-app / persona-pack / persona-journal / `<arbitrary file SoT>` (e.g. `<persona>_presents/` artifact directory)
- Lifecycle monitoring (`triggers_review_of` edges + Daemon scan)
- User-defined custom domains (registered at runtime)

## 3. Architecture (DDD + Hexagonal, BP-aligned v4)

```
┌─[1] Surface              MCP Tool / CLI subcommand
├─[2] Application          Use Cases + SpecRegistry + ProjectionRegistry
├─[3] Domain Core          Graph / CRUD / Compute / Constraint / AutoVersion / Specification
├─[4] Infrastructure       SqliteStorage + RenderingAdapter
├─[5] Daemon               triggers / constraint / workflow / lifecycle scan
├─[6] Adapter (SoT)        outline / persona-pack / mini-app / journal / file-artifact + plugin slot
└─[7] Output channel       mailbox / file / Web (V2) / Tool return
```

### 3.1 Why Specification (BP研究結果)

Per `Specification pattern` (Evans / Fowler / Greg Young):
- First-class composable predicate object
- Supports `and` / `or` / `not` runtime composition
- Domain-neutral (meta-layer, not domain-specific)
- Stored in `SpecRegistry` (Application layer) as named entries

This satisfies the "arbitrary query, dynamic + fixed both" requirement directly without independent invention.

### 3.2 NamedProjection (固定軸)

CQRS Read Model pattern: registered named queries with template binding and target form (Prompt / Markdown / JSON / ASCII). Lives in `ProjectionRegistry` (Application layer).

### 3.3 Rendering (Infrastructure)

Rendering adapter consumes `(Specification, template, target_form)` and produces output text. Template engine selection: see §6 Open question.

### 4.3 depth_limit (確定: 1 hop default)

V2 で `max_depth` 可変化 carry。 P0-P5 は 1 hop で start (= multi-hop は外部 LLM が必要回数 query を回す前提)。

### 4.4 Dump format (確定: JSON-LD primary + frontmatter export)

- **Primary persistence form**: SQLite (`nodes` / `edges` / `versions` table)
- **Portable dump**: JSON-LD (Graph 構造を保持、 外部 graph tool で読み込み可)
- **Export only**: Markdown frontmatter (Obsidian / 静的サイト用)

## 6. Open question (P0 解決推奨)

**Template DSL 選定** — `Rendering` adapter で使う template engine の候補:

| 候補 | PRO | CON | Rating |
|---|---|---|---|
| (a) **Lua snippet** | mlua-mcp-server / mlua-batteries 既存 reuse / DSL 学習コストゼロ / sandbox 軽量 | mlua-mcp-server crate 依存追加 (workspace に組み込む path も検討) | **9/10** [推定: 既存資産流用 + sandbox 性] |
| (b) Tera / Jinja 系 | macro 強い / 学習コスト低い (Django / Jinja2 経験者多い) | 外部 dep 追加、 動的 LL の表現力では Lua より落ちる | 7/10 |
| (c) 自前 minimal lang (Mustache レベル) | dep ゼロ / boundary 明確 | 機能不足が早期に露呈、 後で書き直し risk | 5/10 |

推奨: **(a) Lua snippet**、 Rating 差 +2 以上で AI 決定可能領分。 ただし「mlua-mcp-server crate 依存追加」 を workspace に組み込むか別 path で wire するかは User judgment 領分 (architecture 軸の choice)。

### P2 分割の根拠 (2026-06-15 update)

旧 P2 は `Adapter 3 SoT + wire_doctor + wire_query + 動的 Specification verify` で 1 phase に 5 deliverable 詰め込み = pair-programming 2-3 session 跨ぐ scope。 「vision → 実装」 ではなく 「実装で動かしてから vision を sync する」 evidence-first 規律と整合させて **1 Adapter ずつ land → 観察 → 次 SoT** の Step 形に分割。 P2a の mini-app Adapter は schema が一番素直 (table → Node type、 row.data.{field} → Node properties、 schema.yaml で mapping 規約機械化可) で第 1 件目に適正。

### Tool 命名 rename の根拠 (2026-06-15)

現状 tool name が 2 系統混在で違和感 (旧 `persona-network` → 新 `persona-wire` rename 時の carry):

- lifecycle: `pnet_init` / `pnet_close` (旧 prefix)
- CRUD: `wire_node_create` / `wire_edge_create` / `wire_spec_register` / `wire_projection_register` (新 prefix)

server name = `persona-wire` で MCP 呼出は `mcp__persona-wire__<tool>` になるため `persona_wire_` を tool 側に重複させると冗長。 既存 4 tool は `wire_*` 寄り = `wire_` 統一 (β 案) が diff 最小で natural、 P2-NAMING で集約 land。 v0.1.0 release 前なので breaking 影響ゼロ。
